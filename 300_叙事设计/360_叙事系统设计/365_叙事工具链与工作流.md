# 叙事工具链与工作流

## 一、叙事设计工具链

```
叙事设计需要的工具:

1. 剧本编辑器 (Script Editor)
   功能:
     - 树状分支可视化(节点+连线)
     - 对话逐句编辑，带配音标记
     - 条件触发器编辑(IF player.仁义>60 THEN...)
     - 变量引用({player.name}, {npc.reaction})
     - 实时预览(在编辑器中实际运行对话)
   推荐: 自研工具 or Twine(开源)改

2. 叙事数据库 (Narrative Database)
   存储:
     - 所有对话文本(多语言)
     - 所有模板(程序化叙事模板)
     - 所有变量和标记定义
     - NPC性格参数/关系参数
   格式: JSON/SQLite

3. 叙事调试器 (Narrative Debugger)
   功能:
     - 跳过对话(加速测试)
     - 显示当前所有变量值
     - 显示隐藏选项(调试用)
     - 强制触发特定分支
     - 记录对话历史

4. 分支测试工具 (Branch Tester)
   功能:
     - 自动遍历所有分支(穷举测试)
     - 标记"死路"(不可达的分支)
     - 标记"未写内容"(分支指向空)
     - 统计分支覆盖率
```

## 二、叙事设计师角色分工

```
叙事团队(推荐配置):

叙事总监(1人):
  职责: 整体叙事方向/情感地图/主角弧线/关键场景定稿
  技能: 写作+游戏设计+项目管理

高级叙事设计师(2-3人):
  职责: Lv4/Lv5场景设计/路线剧情/历史事件改编
  技能: 创意写作/历史知识/分支设计
  每人负责: 2条人生路线或2个DLC剧本

叙事系统设计师(1人):
  职责: 程序化叙事系统/模板设计/对话数据库维护
  技能: 写作+基本脚本/逻辑思维

本地化叙事协调(1人):
  职责: 多语言适配/文化差异处理/翻译质量把关
  技能: 多语言/跨文化沟通
```

## 三、叙事设计工作流

```
标准叙事设计流程(一个Lv4任务):

Phase 1: 概念设计(1-2天)
  - 任务主题/情感目标
  - 关键角色/关键场景
  - 分支概图(不详细，只有主要路径)
  → 输出: 一页纸的概念文档

Phase 2: 详细设计(3-5天)
  - 逐场景对话草稿
  - 全部分支定义(每个选项的后果)
  - 触发条件/变量需求
  - NPC角色定位(新NPC or 复用)
  → 输出: 完整剧本草稿+分支图

Phase 3: 内部评审(1天)
  - 叙事团队内部通读
  - 检查: 逻辑一致性/角色一致性/情感节奏/分支合理性
  → 输出: 修改意见列表

Phase 4: 修订(1-2天)
  → 输出: 修订后的剧本

Phase 5: 跨部门评审(1天)
  - 与关卡设计确认(场景可行性)
  - 与程序确认(系统可行性)
  - 与美术确认(资产需求)
  → 输出: 可行性评估+资产需求列表

Phase 6: 最终定稿(1-2天)
  → 输出: 最终剧本+全部对话标记

Phase 7: 实现+测试(随开发周期)
  - 程序实现对话系统
  - 分支测试(遍历所有路径)
  - 情感体验测试(真人试玩→反馈)

总计: 约10-14个工作日/Lv4任务
     Lv5史诗场景需要15-20个工作日
```

## 四、叙事资产命名规范

```
命名规范(避免混乱):

对话文件:
  DIA_[国家]_[角色]_[场景]_[序号]
  例: DIA_QIN_BaiQi_ChangPing_001

任务文件:
  QST_[路线]_[阶段]_[序号]_[名称]
  例: QST_Badao_S2_M03_MingJiangZhiLu

模板文件:
  TPL_[类型]_[子类型]_[情感基调]
  例: TPL_Death_War_Sad

分支文件:
  BRC_[节点名]_[选项数]
  例: BRC_GongGaoZhenZhu_4

日记条目:
  JNL_[日期]_[事件类型]
  例: JNL_BC260_Fall_ChangPing
```

## 五、叙事版本管理

```
使用Git进行叙事版本管理:

原则:
  1. 每个叙事设计师有自己的分支(feature/narrative-XXX)
  2. 完成+评审后合并到develop
  3. 重要叙事节点打tag(如: v0.5-story-lock表示故事锁定)
  
避免冲突:
  - 不同设计师负责不同的角色/路线(避免同时编辑同一NPC)
  - 共用模板/系统文件→需要额外协调
  - 对话数据库使用JSON格式→冲突时人工Merge

变更追踪:
  每次叙事变更记录:
    变更内容 + 变更原因 + 影响范围(哪些分支受影响)
  
  例: "荆轲刺秦场景增加选项D(劝阻荆轲)，
       原因: 玩家反馈缺少'不想让荆轲去死'的选项。
       影响: 荆轲刺秦分支从3选项变为4选项，
            后续剧情需要增加'劝阻成功'的分支内容。"
```
